GPU Architecture & Kernel Files

The CNN library accelerates all operations — convolutional, activation, pooling, and dense (fully-connected) layers — on the GPU using OpenCL. This page explains the overall GPU architecture, the worker model, and how kernel source files are organized.

  1. Unified GPU Architecture
  2. Worker Model
  3. Kernel Source Files
  4. Type System & Precision

Unified GPU Architecture

The CNN library uses a unified GPU execution model where all layers run on OpenCL devices:

  • GPU (OpenCL): All layers — CNN layers (Conv2D, ReLU, Pool, InstanceNorm, BatchNorm, GlobalAvgPool, GlobalDualPool, Residual, Flatten) and dense (fully-connected) layers via ANN::CoreGPUWorker<T>. Bridge kernels copy data between CNN and ANN GPU buffers directly.

This unified GPU execution model is motivated by two observations:

  1. Spatial operations are massively parallel — a single Conv2D layer may have millions of independent multiply-accumulate operations. GPUs excel at this.
  2. Dense layers are already optimized in ANN — rather than re-implementing dense layer logic in OpenCL, the library delegates to the existing, well-tested ANN library. A CoreGPUWorker creates an ANN::CoreGPUWorker<T> that runs dense layers on the same GPU device, with bridge kernels copying data between CNN and ANN GPU buffers directly.
Bridge kernels: After the CNN layers complete (including flatten), the copy_cnn_to_ann kernel copies the flattened activation vector from the CNN activation buffer into the ANN activation buffer on the same GPU. During training, the copy_ann_grad_to_cnn kernel copies the ANN input gradients back into the CNN gradient buffer. No CPU transfer is involved.

Worker Model

The GPU implementation uses a coordinator–worker architecture:

  • CoreGPU<T> is the coordinator. It creates workers, distributes training samples across GPUs, and merges gradients after each epoch. It does not own any GPU device context.
  • CoreGPUWorker<T> is the per-device worker. Each worker owns:
    • An OpenCLWrapper::Core instance bound to one GPU device
    • A GPUBufferManager that allocates and manages all OpenCL buffers
    • A GPUKernelBuilder that generates and registers OpenCL kernels
    • An ANN::CoreGPUWorker<T> instance for dense layer computation on the same GPU device

Each worker can independently process a subset of training samples. During prediction, samples are distributed evenly across workers for maximum throughput.

Kernel Source Files

OpenCL kernel source code is organized into separate files by function. These files are loaded and compiled at runtime by CoreGPUWorker:

FilePurposeKey Kernels
opencl/CNN_Defines.hpp.cl Static header — defines TYPE as float, TILE_SIZE, and ANN types for all kernels TYPE macro, ActvFuncType, Layer
opencl/CNN_Propagate.cpp.cl Forward propagation kernels for ReLU and pooling layers calculate_relu, calculate_maxpool, calculate_avgpool
opencl/CNN_Backpropagate.cpp.cl Gradient computation kernels for ReLU, pooling, and bias gradients calculate_dCost_dRelu, calculate_dCost_dMaxpool, calculate_dCost_dAvgpool, calculate_dCost_dBiases
opencl/CNN_GEMM.cpp.cl Tiled GEMM kernels — convolution forward and backward passes via matrix multiply gemm, gemm_transA, gemm_transB
opencl/CNN_Im2Col.cpp.cl im2col (image-to-column) transformation for GEMM-based convolution im2col, col2im
opencl/CNN_Update.cpp.cl Parameter update, gradient accumulation, and buffer zeroing zero_buffer, accumulate_gradients, update_parameters, update_parameters_adam
opencl/CNN_Normalization.cpp.cl Normalization kernels (InstanceNorm and BatchNorm, unified via N and sampleStride params) norm_compute_mean, norm_compute_var, norm_normalize, norm_dGammaBeta, norm_dInput, norm_update_running_stats
opencl/CNN_GlobalAvgPool.cpp.cl Global average pooling kernels gap_propagate, gap_backpropagate
opencl/CNN_GlobalDualPool.cpp.cl Global dual pooling kernels (max + avg concatenated) gdp_propagate, gdp_backpropagate
opencl/CNN_Bridge.cpp.cl Bridge kernels — copy data between CNN and ANN GPU buffers on the same OpenCL core copy_cnn_to_ann, copy_ann_grad_to_cnn
opencl/CNN_Residual.cpp.cl Residual (skip connection) kernels — identity addition and 1×1 projection residual_add, residual_add_proj, residual_bwd, residual_bwd_proj_dskip, residual_bwd_proj_dw

Type System & Precision

All OpenCL kernels use the TYPE macro defined in CNN_Defines.hpp.cl. This file is a static header included by all kernel sources:

// CNN_Defines.hpp.cl — static header file included by all kernels
#define TYPE float
#define TILE_SIZE 16 // Work-group tile dimension for GEMM kernels

// ANN types (must match C++ ANN::ActvFuncType and ANN::Layer)
typedef enum { ACTV_RELU = 0, ACTV_SIGMOID = 1, ACTV_TANH = 2, ACTV_SOFTMAX = 3, ACTV_UNKNOWN = 4 } ActvFuncType;

typedef struct {
    ulong numNeurons;
    ActvFuncType actvFuncType;
} Layer;

CNN_Defines.hpp.cl is loaded once via addSourceFile() during kernel compilation. It defines TYPE as float and is included by all other kernel source files. The library currently supports only float precision; double and int instantiations of Core<T> are not supported for GPU mode.

Double precision on GPU: Not all GPUs support cl_khr_fp64 (double precision). The library currently only supports float for GPU mode. For maximum compatibility, use float.

Kernel Builder Pattern

Rather than hardcoding kernel launches, the library uses a GPUKernelBuilder that walks the layer configuration and dynamically generates the correct sequence of kernel calls:

  1. Walk the layer list — iterate over layersConfig.cnnLayers in order
  2. Track shapes — compute output shapes layer-by-layer, maintaining activation buffer offsets
  3. Register kernels — for each layer, register the appropriate kernel with its arguments (buffer references, offsets, dimensions)
  4. Execute in sequence — at runtime, kernels execute in the order they were registered

This pattern means adding a new layer type requires only:

  • Writing the OpenCL kernel (in the appropriate .cl file)
  • Adding a case in GPUKernelBuilder to register it
  • Adding buffer allocation in GPUBufferManager if the layer needs new buffers
Why not a single monolithic kernel? Separate kernels per layer allow the OpenCL runtime to optimize each kernel independently, enable better GPU occupancy (each kernel has different work-item counts), and make debugging easier (you can test each kernel in isolation).
