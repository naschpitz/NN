Convolution

The 2D convolution operation is the fundamental building block of CNNs. This page covers the forward pass, padding & stride, and backpropagation through convolutional layers.

2D Convolution

Given an input tensor and a set of learnable filters, convolution produces an output by sliding each filter across the input and computing element-wise products.

Mathematical Definition

\[ \text{output}[f, o_h, o_w] = \text{bias}[f] + \sum_{c=0}^{C-1} \sum_{k_h=0}^{K_H-1} \sum_{k_w=0}^{K_W-1} \text{input}[c, o_h \cdot s_y + k_h - p_y, o_w \cdot s_x + k_w - p_x] \cdot \text{filter}[f, c, k_h, k_w] \]

Index Notation

  • f — Filter index (0 to F-1), determines the output channel
  • c — Input channel index (0 to C-1)
  • o_h, o_w — Output spatial position (height, width)
  • k_h, k_w — Filter kernel position (height, width)
  • s_y, s_x — Stride in vertical and horizontal directions
  • p_y, p_x — Padding applied to input (height, width)
  • K_H, K_W — Filter kernel dimensions
  • C — Number of input channels
  • F — Number of filters (output channels)
The input is padded with zeros before convolution. The term o_h · s_y + k_h - p_y computes the actual input index by accounting for stride, kernel position, and padding offset.
Code: Conv2D<T>::propagate(input, config, params) in CNN_Conv2D.cpp implements the full 6-nested-loop convolution (CPU path). On GPU: convolution is computed via im2col + GEMM — the im2col kernel in opencl/CNN_Im2Col.cpp.cl rearranges input patches, then the gemm kernel in opencl/CNN_GEMM.cpp.cl performs the matrix multiply.

Padding & Stride

Padding and stride control the spatial dimensions of the output. The output size formula is:

\[ O_H = \left\lfloor \frac{H + 2P_Y - K_H}{S_Y} \right\rfloor + 1 \]

Similarly for width: \(O_W = \left\lfloor \frac{W + 2P_X - K_W}{S_X} \right\rfloor + 1\)

Padding Modes

ModePadding ValueFormulaEffect
VALIDP = 0\(O_H = \lfloor \frac{H - K_H}{S_Y} \rfloor + 1\)Output shrinks; no padding
SAMEP = ⌊K/2⌋\(O_H = \lceil \frac{H}{S_Y} \rceil\)Preserves spatial dims when stride=1
FULLP = K - 1\(O_H = H + K_H - 1\)Output expands; maximum padding
Code: SlidingStrategy::computePadding(kernelSize, strategy) in CNN_SlidingStrategy.hpp returns 0 (VALID), ⌊K/2⌋ (SAME), or K−1 (FULL). The output dimensions are computed inside Conv2D<T>::propagate() using the formula above.

Backpropagation through Conv2D

During backpropagation, gradients flow through the convolution layer in reverse. Given the upstream gradient \(\frac{\partial C}{\partial O[f,o_h,o_w]}\), we compute gradients with respect to filters, biases, and inputs.

Filter Gradient

\[ \frac{\partial C}{\partial W[f,c,k_h,k_w]} = \sum_{o_h} \sum_{o_w} \frac{\partial C}{\partial O[f,o_h,o_w]} \cdot X[c, o_h \cdot s_y + k_h - p_y, o_w \cdot s_x + k_w - p_x] \]
Code: Inside Conv2D<T>::backpropagate() in CNN_Conv2D.cpp — the line dFilters[filterIdx] += dOutVal * input.at(c, ih, iw). On GPU: im2col + gemm_transB — computes dFilters = dOut × im2col(input)T.

Bias Gradient

\[\frac{\partial C}{\partial b[f]} = \sum_{o_h} \sum_{o_w} \frac{\partial C}{\partial O[f,o_h,o_w]}\]
Code: Inside Conv2D<T>::backpropagate() — the line dBiases[f] += dOutVal. On GPU: calculate_dCost_dBiases kernel.

Input Gradient

\[\frac{\partial C}{\partial X[c,i_h,i_w]} = \sum_f \sum_{k_h} \sum_{k_w} \frac{\partial C}{\partial O[f,o_h,o_w]} \cdot W[f,c,k_h,k_w]\]

where \(o_h = \frac{i_h + p_y - k_h}{s_y}\) (only when evenly divisible).

Code: Inside Conv2D<T>::backpropagate() — the line dInput.at(c, ih, iw) += dOutVal * params.filterAt(f, c, kh, kw). On GPU: gemm_transA + col2im — computes dInput = col2im(FiltersT × dOut).
The input gradient is essential when stacking multiple convolutional layers, since it propagates the error signal to previous layers.

GPU Implementation: im2col + GEMM

On the GPU, convolution is not computed directly using the formulas above. Instead, it is reformulated as a matrix multiplication using the im2col (image-to-column) transformation. This is the same approach used by cuDNN, MKL-DNN, and most production deep learning frameworks.

Why Not Direct Convolution?

Direct convolution — one work-item per output element, each computing a nested sum over channels and filter positions — has poor GPU performance because:

  • Scattered memory access: Each work-item reads non-contiguous input locations (jumping by stride, across channels). Adjacent work-items read different, non-adjacent memory addresses, defeating GPU memory coalescing.
  • No data reuse: Overlapping receptive fields mean the same input elements are loaded from global memory many times by different work-items, wasting bandwidth.
  • Low arithmetic intensity: The ratio of computation to memory access is low, making the kernel memory-bound on most GPUs.

Why im2col Is Needed

The input is a 3D tensor (C, H, W) stored flat in row-major memory. The values a filter needs for a single output position are scattered across memory — they are not contiguous. Consider a 3×3 filter at output position (oh=5, ow=10) on a (3, 450, 600) input. The filter needs these 27 values:

input[0, 5, 10]  input[0, 5, 11]  input[0, 5, 12]    ← channel 0, row 5
input[0, 6, 10]  input[0, 6, 11]  input[0, 6, 12]    ← channel 0, row 6
input[0, 7, 10]  input[0, 7, 11]  input[0, 7, 12]    ← channel 0, row 7
input[1, 5, 10]  input[1, 5, 11]  input[1, 5, 12]    ← channel 1, row 5
...                                                    ← 9 more values

In flat memory (row-major, C×H×W), those are at positions:

3010, 3011, 3012, 3610, 3611, 3612, 4210, 4211, 4212,   ← channel 0
273010, 273011, 273012, ...                               ← channel 1 (offset by 450×600 = 270,000)

They jump by 600 between rows and by 270,000 between channels. This is terrible for GEMM, which needs contiguous rows and columns to stream through memory efficiently.

The im2col Transformation

im2col gathers those scattered values and writes them as one contiguous column. It does this for every output position, producing a column matrix:

\[ \underbrace{\text{Input}}_{\scriptsize (C, H, W)} \xrightarrow{\text{im2col}} \underbrace{\text{Column Matrix}}_{\scriptsize (C \cdot K_H \cdot K_W, \; O_H \cdot O_W)} \]

Column j of the result contains the \(C \cdot K_H \cdot K_W\) input values needed for output position j, packed contiguously. Now GEMM can stream through memory sequentially.

The trade-off is memory: im2col duplicates data because adjacent patches share overlapping values. For a 3×3 filter with stride 1, each input value gets copied into up to 9 columns. But the regular access pattern makes GEMM dramatically faster than reading scattered locations.

1×1 convolutions: When the filter is 1×1, the "patch" is just one pixel across all channels — which is already a column of the input reshaped as (C, H×W). Frameworks like PyTorch skip im2col entirely for 1×1 convolutions. This library does not currently optimize for that case.

Convolution as Matrix Multiply

With the input rearranged, convolution becomes a single matrix multiplication:

\[ \underbrace{\text{Output}}_{\scriptsize (F, \; O_H \cdot O_W)} = \underbrace{\text{Filters}}_{\scriptsize (F, \; C \cdot K_H \cdot K_W)} \times \underbrace{\text{im2col(Input)}}_{\scriptsize (C \cdot K_H \cdot K_W, \; O_H \cdot O_W)} + \text{Bias} \]

Matrix multiplication has perfectly regular access patterns: each element of the output is a dot product of a row of A and a column of B. GPUs can exploit this with tiled GEMM — loading tiles into fast local (shared) memory and reusing each element \(T\) times (where \(T\) is the tile size), reducing global memory traffic by ~\(T\times\).

Tiled GEMM

The GEMM kernel divides the output matrix into \(T \times T\) tiles (currently \(T = 16\)). Each work-group computes one tile:

  1. All work-items collaboratively load one tile of A and one tile of B from global memory into local memory
  2. A barrier() ensures the tile is fully loaded
  3. Each work-item computes a partial dot product using only local memory (fast)
  4. A second barrier() ensures the tile is consumed before loading the next
  5. Repeat for all tiles along the K dimension, accumulating results
  6. Write the final accumulated value to global memory
The tiled approach loads each element from global memory once and reuses it 16 times (for TILE_SIZE=16), reducing memory bandwidth requirements by ~16×. This transforms the kernel from memory-bound to compute-bound on most GPUs.

Backpropagation with GEMM

The backward pass reuses the same GEMM machinery. Each gradient computation is a matrix multiply between the upstream gradient (dOut) and the appropriate matrix — filters or im2col(input) — with the correct one transposed. The "transA" and "transB" variants don't actually transpose data in memory; they simply change the indexing during tile loading to read rows as columns or vice versa.

What We ComputeMeaningFormulaKernel
dFilters How much each filter weight contributed to the error \(\nabla W = \text{dOut} \times \text{im2col(Input)}^T\) gemm_transB
dInput How to distribute the error back to the previous layer \(\nabla X = \text{col2im}(\text{Filters}^T \times \text{dOut})\) gemm_transA + col2im
dBiases How much each bias contributed to the error \(\nabla b[f] = \sum_{o_h, o_w} \text{dOut}[f, o_h, o_w]\) calculate_dCost_dBiases

Why transB for dFilters?

dFilters asks: "for each filter weight, how much did it contribute to the total error?" The answer is the correlation between the upstream gradient (dOut, one value per output position) and the input patches (im2col, one column per output position). This is dOut × im2colT — we transpose im2col because it is stored as (filterVolume, spatialPositions) but we need to correlate along the spatial dimension, so the spatial positions must be the shared K dimension.

Why transA for dInput?

dInput asks: "how should the error be distributed back to each input pixel?" The answer requires projecting dOut through the filters in reverse — FiltersT × dOut. We transpose the Filters matrix because it is stored as (numFilters, filterVolume) but we need (filterVolume, numFilters) so that we map from output-space errors back into input-space (filterVolume) positions. The result is in im2col layout, so col2im scatters it back to the actual input tensor shape, accumulating where patches overlap.

col2im

col2im is the inverse of im2col: it scatters the column matrix back into the input tensor shape. Since overlapping patches map multiple column entries to the same input position, col2im accumulates (sums) contributions. The gradient buffer must be zeroed before col2im runs.

To avoid float atomics (not reliably available in OpenCL), col2im uses one work-item per input element \((c, i_h, i_w)\). Each work-item iterates over all output positions whose receptive field includes that input element and sums the corresponding column values.

Code: The GEMM kernels are in opencl/CNN_GEMM.cpp.cl, im2col/col2im in opencl/CNN_Im2Col.cpp.cl. The kernel wiring is in CNN_GPUKernelBuilder.cpp — see the CONV cases in addPropagateKernels() and addBackpropagateKernels(). The im2col workspace buffer (cnn_im2col) is allocated in CNN_GPUBufferManager.cpp.

Performance: CNN Library vs PyTorch

On an NVIDIA RTX 4090, this library's GPU convolution is roughly 7–10× slower than PyTorch/cuDNN for the same network architecture. This is not a single bottleneck — it is the product of several independent factors:

Tensor Cores — ~2×

The RTX 4090 has dedicated tensor cores that deliver 165 TFLOPS (TF32) vs 83 TFLOPS on regular CUDA cores. PyTorch uses TF32 by default — a slightly reduced precision format (10-bit mantissa vs 23-bit for FP32) that is accurate enough for training but runs at 2× throughput.

Tensor cores are only accessible via CUDA's WMMA instructions or through cuBLAS/cuDNN. There is no OpenCL API for tensor cores. This is a hard 2× ceiling that cannot be broken on NVIDIA hardware with OpenCL.

Winograd Algorithm for 3×3 Convolutions — ~2×

cuDNN uses the Winograd minimal filtering algorithm for 3×3 convolutions, which is a mathematical transformation that reduces the number of multiplications by ~2.25×. Instead of 9 multiplies per output position per channel, Winograd does ~4. This is not a GPU trick — it is a different algorithm entirely that could be implemented in OpenCL, but it is complex (requires pre/post-transforming both filters and input tiles into the Winograd domain).

cuBLAS Hand-Tuned GEMM vs Textbook Tiled GEMM — ~1.5–2×

cuBLAS GEMM kernels are hand-tuned per GPU architecture by NVIDIA engineers over many years. They use register blocking, vectorized loads, software pipelining, and warp-level shuffle instructions — all CUDA-specific features. This library's tiled GEMM is correct and uses shared memory for data reuse, but it likely hits 20–30% of peak FLOPS while cuBLAS hits 80–90%.

CUDA vs OpenCL Driver Overhead — ~1.2–1.3×

NVIDIA's OpenCL implementation is a second-class citizen compared to CUDA. The CUDA runtime has lower kernel dispatch latency, better compiler optimizations, and tighter hardware integration.

Combined Effect

These factors multiply:

\[ 2\times \text{(tensor cores)} \times 2\times \text{(Winograd)} \times 1.5\times \text{(cuBLAS)} \times 1.2\times \text{(CUDA overhead)} \approx 7\times \]

The dominant factors are tensor cores and Winograd — hardware that cannot be accessed from OpenCL and an algorithm not currently implemented. The GEMM quality gap matters but is the smallest factor.

On AMD GPUs (e.g., RX 480), the tensor core gap disappears since AMD's matrix cores are not used by PyTorch's ROCm backend for all operations. The gap narrows to roughly 3–4× (Winograd + hand-tuned GEMM + framework overhead).
