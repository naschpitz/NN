Convolution Kernel Pipeline

Convolution on the GPU is computed via im2col + GEMM instead of direct convolution kernels. The im2col transformation rearranges the input so that convolution becomes a matrix multiplication — see Math: im2col + GEMM for the full explanation. The tables below show exactly which kernels run for each logical operation per conv layer:

Forward Pass (per conv layer)

StepLogical OperationKernelsWhat It Does
1 Compute convolution output im2col → gemm Rearranges input patches into a column matrix, then computes Output = Filters × im2col(Input) + Bias. Each row of the Filters matrix is one filter flattened; each column of the im2col matrix is one input patch. The matrix product computes all filters at all spatial positions in one operation.

Backward Pass (per conv layer)

StepLogical OperationKernelsWhat It Does
1 Compute filter gradients im2col → gemm_transB Computes dFilters = dOut × im2col(Input)T. Each element of dFilters is the dot product of one row of dOut (gradient at each spatial position for filter f) and one column of im2colT (input values seen by that filter element). B is transposed because im2col is stored as (rows, cols) but we need the input patches as rows, not columns.
2 Compute bias gradients calculate_dCost_dBiases Sums dOut over all spatial positions per filter: dBias[f] = Σ dOut[f, o_h, o_w]. Uses parallel reduction (one work-group per filter).
3 Compute input gradients gemm_transA → col2im Computes dInput_cols = FiltersT × dOut, then scatters back to input shape. A is transposed because Filters are stored as (numFilters, filterVolume) but we need (filterVolume, numFilters) to project the output gradients back into the input space. The col2im kernel reverses the im2col rearrangement, accumulating overlapping patch contributions.
Why transpose variants instead of actually transposing matrices? Transposing a large matrix on GPU is itself an expensive memory operation. Instead, gemm_transA and gemm_transB read the matrix elements in transposed order during the tile-loading phase — A[col][row] instead of A[row][col]. Same data, different indexing, zero extra memory or kernel launches.

GEMM Kernels

CNN_GEMM.cpp.cl

Tiled General Matrix Multiply kernels. All three variants use the same tiled algorithm with local memory — only the indexing into A or B differs for transpose variants.

Dispatch: 2D global work size (ceil(N/TILE)*TILE, ceil(M/TILE)*TILE), local (TILE, TILE).

KernelMatrix OperationConvolution Equivalent
gemm C = A × B + bias Forward: compute all filter outputs at all spatial positions
gemm_transB C = A × BT Backward: compute how much each filter weight contributed to the error (dFilters)
gemm_transA C = AT × B Backward: project output error back into input space (dInput)

The transpose variants (gemm_transA, gemm_transB) have the same arguments except without bias and offsetBias.

im2col / col2im Kernels

CNN_Im2Col.cpp.cl

im2col

Rearranges input patches into a column matrix for GEMM-based convolution. One work-item per output element. Dispatch: 1D, nElements = C_in × K_H × K_W × O_H × O_W.

ArgumentTypeDescription
inputglobal TYPE*Input activation buffer
colglobal TYPE*Output column matrix (cnn_im2col)
inputOffsetulongOffset into input buffer
C, H, WulongInput tensor dimensions
kH, kWulongFilter dimensions
strideY, strideXulongConvolution strides
padY, padXulongPadding amounts
outH, outWulongOutput spatial dimensions

col2im

Inverse of im2col — scatters column matrix back to input tensor shape, accumulating overlapping contributions. Used during backpropagation to convert dInput from column layout back to tensor layout. One work-item per input element (avoids float atomics). Dispatch: 1D, nElements = C × H × W.

The gradient buffer must be zeroed before calling col2im, since it accumulates contributions from overlapping patches.

Arguments are the same as im2col, except col is the input (column data) and output is the gradient buffer being written to.
