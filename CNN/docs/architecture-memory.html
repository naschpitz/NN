Memory Layout

Understanding how the CNN library stores data in memory is essential for reasoning about performance, debugging numerical issues, and extending the library. This page explains the NCHW tensor layout, how parameters are stored, and the implications for GPU computation.

NCHW vs. NHWC

There are two common ways to lay out multi-dimensional image data in memory:

  • NCHW (channels-first): All spatial values for channel 0 come first, then all values for channel 1, etc. This is what the CNN library uses.
  • NHWC (channels-last): At each spatial position, all channel values are stored together. This is used by TensorFlow and some other frameworks.

For a 3-channel, 2×2 image with RGB values, the difference is:

LayoutMemory Order
NCHW[R₀₀, R₀₁, R₁₀, R₁₁, G₀₀, G₀₁, G₁₀, G₁₁, B₀₀, B₀₁, B₁₀, B₁₁]
NHWC[R₀₀, G₀₀, B₀₀, R₀₁, G₀₁, B₀₁, R₁₀, G₁₀, B₁₀, R₁₁, G₁₁, B₁₁]
Why NCHW? Convolution processes one channel at a time — it slides a 2D filter over the 2D spatial grid of a single channel, then moves to the next channel. With NCHW, the spatial values being convolved are contiguous in memory, which maximizes CPU cache hits and GPU memory coalescing. With NHWC, the convolution would need to "skip" across channels at each spatial position, causing poor cache behavior.

Address Computation

Given a 3D tensor with shape \((C, H, W)\), the flat array index for element \((c, h, w)\) is:

\[ \text{index}(c, h, w) = c \times H \times W + h \times W + w \]

This formula is used throughout the library and in GPU kernels. For a concrete example with shape (2 channels, 3 rows, 4 columns):

IndexElementDescription
0tensor[0,0,0]Channel 0, Row 0, Col 0
1tensor[0,0,1]Channel 0, Row 0, Col 1
2tensor[0,0,2]Channel 0, Row 0, Col 2
3tensor[0,0,3]Channel 0, Row 0, Col 3
4tensor[0,1,0]Channel 0, Row 1, Col 0
......(continues row by row)
11tensor[0,2,3]Channel 0, Row 2, Col 3 (last element of channel 0)
12tensor[1,0,0]Channel 1, Row 0, Col 0 (first element of channel 1)
......(continues for channel 1)
23tensor[1,2,3]Channel 1, Row 2, Col 3 (last element)

The total number of elements is \(C \times H \times W = 2 \times 3 \times 4 = 24\).

Tensor Storage

The Tensor3D<T> class is the fundamental data structure. It stores data as a flat std::vector<T> paired with a Shape3D struct containing channels, height, and width. This design has several important properties:

  • Zero-cost flattening — since the underlying storage is already a 1D vector, the Flatten layer can hand it directly to ANN's dense layers without any data copying or reallocation.
  • Efficient resizing — when a layer changes the tensor's shape (e.g., after pooling reduces spatial dimensions), only the shape metadata is updated if the new size fits within the existing vector's capacity. No reallocation occurs until the tensor grows beyond its current capacity.
  • Direct data access — the .data() method returns a raw pointer to the underlying array. This is used by GPU code to efficiently transfer data between host and device memory.
  • Element access — the at(c, h, w) method computes the flat index and returns a reference, providing a convenient 3D interface over the flat storage.

Filter Storage

Convolution filter weights for all filters in a layer are stored in a single contiguous flat vector. The layout follows a 4D indexing scheme: (filter, inputChannel, kernelRow, kernelCol):

\[ \text{index}(f, c, kh, kw) = f \times C_{in} \times K_h \times K_w + c \times K_h \times K_w + kh \times K_w + kw \]

For a layer with \(F = 2\) filters, \(C_{in} = 3\) input channels, and \(3 \times 3\) kernels, the total filter storage is \(2 \times 3 \times 3 \times 3 = 54\) elements. All of filter 0's weights come first (27 elements), then all of filter 1's weights (27 elements).

Biases are stored separately as a 1D vector of length \(F\) — one bias per filter.

Normalization Parameter Storage

Each InstanceNorm layer stores four vectors, each of length \(C\) (number of channels):

VectorPurposeUpdated By
gammaLearnable scale factor per channelGradient descent
betaLearnable shift factor per channelGradient descent
runningMeanRunning average of channel meansExponential moving average
runningVarRunning average of channel variancesExponential moving average

All four vectors are indexed by channel: gamma[c], beta[c], etc. They are stored per-layer in the Parameters struct.

Residual Projection Storage

When a residual block uses a projection shortcut (channel count changes), the projection parameters are stored as:

  • Weights — a flat vector of \(C_{out} \times C_{in}\) elements, representing a 1×1 convolution. Indexed as \(W[oc \times C_{in} + ic]\).
  • Biases — a vector of \(C_{out}\) elements, one per output channel.

GPU Buffer Layout

On the GPU, the same NCHW layout is used in OpenCL buffers. The GPUBufferManager allocates flat OpenCL buffers that mirror the CPU tensor layout. Key buffer categories:

  • Activation buffers — store layer outputs (the "activation" at each layer). These are reused across layers during propagation.
  • Gradient buffers — store the gradient at each layer during backpropagation. Also reused.
  • Parameter buffers — store filter weights, biases, normalization parameters, and residual projection parameters. These persist across training steps.
  • Accumulator buffers — store accumulated gradients (dFilters, dBiases, dGamma, dBeta, dW, dBias) across samples within an epoch.
  • Auxiliary buffers — max pool indices, normalization intermediate values (mean, variance, normalized activations), residual skip input snapshots.
Buffer offsets: Rather than allocating separate buffers for each layer, the library uses a single large buffer per category (e.g., one big filter buffer for all conv layers). Each layer is assigned an offset within this buffer. This reduces OpenCL memory allocation overhead and simplifies buffer management.
