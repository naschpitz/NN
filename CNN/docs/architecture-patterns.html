Design Patterns

The CNN library employs several well-known software design patterns to achieve flexibility, extensibility, and clean separation of concerns. Understanding these patterns is valuable both for using the library effectively and for extending it with new layer types or backends.

  1. Factory Pattern
  2. ANN Delegation Pattern
  3. Stateless Layer Pattern
  4. Template Pattern
  5. Extending the Library

🏭 Factory Pattern

Problem: The library supports two completely different execution backends — CPU and GPU. Client code (like NN-CLI) should not need to know which backend is in use, nor should it need different code paths for each backend.

Solution: Core<T>::makeCore(const CoreConfig<T>& config) is a static factory method that examines config.deviceType and returns a std::unique_ptr<Core<T>> pointing to either CoreCPU<T> or CoreGPU<T>. The caller interacts only with the abstract Core<T> interface.

// Client code — works identically with CPU or GPU
auto core = Core<float>::makeCore(config);

// Train — the factory already chose the right backend
core->train(samples.size(), CNN::makeSampleProvider(samples));

// Predict — same interface regardless of backend
auto result = core->predict(input);

Benefit: Switching between CPU and GPU requires only changing config.deviceType from "cpu" to "gpu" in the JSON configuration. Zero code changes. The factory handles all the complexity of creating the appropriate backend with its buffers, kernels, and workers.

🔗 ANN Delegation Pattern

Problem: A complete CNN pipeline needs dense (fully-connected) layers for the final classification/regression. Re-implementing dense layers inside the CNN library would duplicate the existing, well-tested ANN library.

Solution: Both CoreCPU<T> and CoreGPUWorker<T> contain an ANN::Core<T> instance. After CNN layers process the input and flatten it to a 1D vector, this vector is handed to ANN for dense layer processing. ANN handles its own propagation, backpropagation, and parameter updates.

// In CoreCPU::predict()
// 1. CNN layers process the image
Tensor3D<T> cnnOut = propagateCNN(input);

// 2. Flatten to 1D
Tensor1D<T> flatInput = Flatten<T>::propagate(cnnOut);

// 3. Hand off to ANN for dense layers
ANN::Input<T> annInput(flatInput.begin(), flatInput.end());
ANN::Output<T> annOutput = annCore->predict(annInput);

Benefit: No code duplication. The ANN library's dense layer implementation — including activation functions (ReLU, sigmoid, softmax, etc.), cost functions (MSE, cross-entropy), and optimizers — is reused directly. Bug fixes and improvements to ANN automatically benefit CNN.

Trade-off: During GPU training, data must be transferred between GPU (CNN) and CPU (ANN) at each sample. This transfer is small (the flattened vector is typically hundreds to thousands of elements) and does not significantly impact performance.

📦 Stateless Layer Pattern

Problem: Traditional OOP would make each layer an object with internal state (weights, activations, gradients). This creates ownership issues, makes threading harder, and tightly couples layers to specific positions in the network.

Solution: All CNN layer classes (Conv2D, ReLU, Pool, InstanceNorm, Flatten, GlobalAvgPool, GlobalDualPool, Residual) contain only static methods. They have no instance state. Everything needed — input tensor, configuration, parameters — is passed as function arguments.

// No object creation needed — just call the static method
static Tensor3D<T> Conv2D<T>::propagate(
  const Tensor3D<T>& input,       // Input tensor
  const ConvLayerConfig& config,    // Layer configuration
  const ConvParameters<T>& params); // Learned parameters

Benefits:

  • Thread-safe — no shared mutable state means multiple threads can call the same layer function concurrently without locks.
  • Composable — layers can be chained in any order without worrying about initialization order or internal state dependencies.
  • Testable — each layer function can be tested in isolation by passing specific inputs and checking outputs. No setup ceremony required.
  • Flexible parameter management — parameters live in the Core classes, not in layers. This makes serialization, gradient accumulation, and multi-GPU synchronization much simpler.

🎯 Template Pattern

Problem: Different applications need different numeric precision — float for speed, double for accuracy, int for quantized inference. Writing separate implementations for each type is error-prone and unmaintainable.

Solution: All classes are C++ templates parameterized on the numeric type T. The template parameter flows through the entire class hierarchy and all function signatures.

template <typename T>
class Core { ... };

template <typename T>
class Conv2D {
  static Tensor3D<T> propagate(const Tensor3D<T>& input, ...);
};

// Instantiate for different types
auto floatCore = Core<float>::makeCore(config);   // Fast, lower precision
auto doubleCore = Core<double>::makeCore(config); // Slower, higher precision

Benefits:

  • Type safety — the compiler catches type mismatches at compile time. You can't accidentally mix float weights with double activations.
  • Zero overhead — templates generate specialized code for each type at compile time. No runtime type checking or virtual dispatch.
  • Single source of truth — one implementation handles all numeric types. Bug fixes apply everywhere.

For GPU: The template parameter also drives the OpenCL #define TYPE macro, ensuring GPU kernels use the same precision as the host code.

Extending the Library

Thanks to these patterns, adding a new layer type is straightforward:

  1. Create a new stateless layer class with static propagate() and backpropagate() methods.
  2. Add a case in CoreCPU::propagateCNN() and CoreCPU::backpropagateCNN().
  3. Write OpenCL kernels in the appropriate .cl file.
  4. Register the kernel in GPUKernelBuilder.
  5. Add buffer allocation in GPUBufferManager if the layer needs new GPU buffers.
  6. Add the layer type to the JSON parser in the configuration module.

The factory pattern ensures no changes to client code. The stateless pattern ensures the new layer integrates cleanly. The template pattern ensures it works with all numeric types.
