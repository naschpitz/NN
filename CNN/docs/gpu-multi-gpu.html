Multi-GPU Training & Kernel Compilation

The CNN library supports training across multiple GPUs simultaneously. This page explains how work is distributed, how gradients are merged, and how OpenCL kernels are compiled.

  2. Gradient Merging

  5. Kernel Compilation

Data-Parallel Training

Multi-GPU training uses data parallelism: the same model is replicated on each GPU, and the training data is split across devices. Each GPU processes a subset of the batch independently, then gradients are merged before updating parameters.

The key advantage of data parallelism is its simplicity — the model is identical on each GPU, so no model partitioning is needed. Training throughput scales nearly linearly with the number of GPUs, since the computational bottleneck (convolution) runs independently on each device.

Gradient Merging

After each epoch, gradients from all workers are merged on the CPU. The process for CNN gradients is:

  1. Read gradients — each worker reads its accumulated gradient buffers (dFilters, dBiases, normalization dGamma/dBeta, residual projection dW/dBias) from GPU memory to CPU memory via readAccumulatedGradients().
  2. Sum across workers — CoreGPU::mergeCNNGradients() sums the gradient arrays element-wise from all workers into the first worker's accumulators.
  3. Write back — the merged gradients are written back to each worker's GPU buffers via setAccumulators().

For ANN (dense layer) parameters, a different approach is used:

  1. Each worker's ANN core independently updates its own dense parameters during training.
  2. After each epoch, CoreGPU::mergeANNParameters() averages the dense layer weights and biases across all workers.
  3. The averaged parameters are written back to every worker's ANN core.
Why sum CNN gradients but average ANN parameters? CNN gradients are accumulated (not applied yet), so summing gives the total gradient across all samples. ANN parameters have already been updated independently on each worker, so averaging brings them back to a consensus value.

Parameter Synchronization

After gradient merging and parameter update, updated CNN parameters (filter weights, biases, normalization gamma/beta, residual projection weights/biases) are broadcast from the first worker back to all other workers. This ensures every GPU starts the next epoch with identical parameters.

The synchronization happens at the end of each epoch, not after each sample or mini-batch. This minimizes expensive GPU↔CPU data transfers while still maintaining parameter consistency across devices.

Multi-GPU Training Loop

Here is the complete multi-GPU training loop as implemented in CoreGPU::train():

for epoch in [0, numEpochs):
  shuffle(samples)

  // 1. Each worker trains on its subset in parallel
  for each worker in workers:  // launched as threads
    worker.trainSubset(samples, startIdx, endIdx, ...)

  // 2. Read CNN gradient accumulators from each GPU
  for each worker:
    worker.readAccumulatedGradients()

  // 3. Sum CNN gradients across all workers
  mergeCNNGradients()

  // 4. Write merged gradients back & run SGD/Adam update kernels
  for each worker:
    worker.setAccumulators()
    worker.update(totalSamples)

  // 5. Average ANN parameters across all workers
  mergeANNParameters()

  // 6. Sync updated parameters to all workers
  // (done implicitly via shared parameter references)

Kernel Compilation

OpenCL kernels are compiled at runtime when a CoreGPUWorker is first created. The compilation process:

  1. Write type definitions — generate CNN_Defines.hpp.cl with #define TYPE float (or double) based on the template parameter T.
  2. Load source files — read all .cl kernel source files from the opencl/ directory.
  3. Compile — submit all source to the OpenCL runtime for compilation on the target GPU device. Compilation errors are caught and reported with the full OpenCL build log.
  4. Register kernels — GPUKernelBuilder walks the network architecture and registers the specific kernels needed, with their buffer arguments and work dimensions.

Compilation happens only once per worker. Once compiled, kernels are reused across all training epochs and prediction calls.

Compilation time: OpenCL kernel compilation can take several seconds, especially for complex networks with many layers. This is a one-time cost at initialization. If you see a delay when creating a Core<T> with GPU mode, this is the kernel compilation step.

Configuration

Multi-GPU training is controlled by the numGPUs field in CoreConfig:

ValueBehavior
0Use all available GPUs (default). The library queries OpenCL for all GPU devices and creates one worker per device.
1Use a single GPU. No gradient merging overhead — simplest configuration.
NUse exactly N GPUs. If fewer than N devices are available, an error is raised.

JSON Configuration

{
  "deviceType": "gpu",
  "numGPUs": 0,        // 0 = all available GPUs
  "numGPUs": 2,        // Use exactly 2 GPUs
  ...
}
Performance tip: For small datasets, a single GPU may be faster than multiple GPUs because gradient merging has overhead. Multi-GPU training shines when the dataset is large enough that the per-GPU computation time dominates the merge time.
