GPU Architecture & Kernel Files

The CNN library accelerates convolutional, activation, and pooling operations on the GPU using OpenCL. Dense (fully-connected) layers remain on the CPU, handled by the ANN library. This page explains the overall GPU architecture, the worker model, and how kernel source files are organized.

Hybrid GPU/CPU Design

The CNN library uses a hybrid execution model where different parts of the neural network run on different hardware:

  • GPU (OpenCL): All CNN layers — Conv2D, ReLU, Pool, InstanceNorm, BatchNorm, GlobalAvgPool, GlobalDualPool, Residual, and Flatten. These operations are highly parallel and benefit enormously from GPU throughput.
  • CPU (ANN library): Dense (fully-connected) layers. After the CNN layers produce a flattened 1D vector, the data is transferred back to CPU memory where the ANN library handles dense layer propagation and backpropagation.

This split is motivated by two observations:

  1. Spatial operations are massively parallel — a single Conv2D layer may have millions of independent multiply-accumulate operations. GPUs excel at this.
  2. Dense layers are already optimized in ANN — rather than re-implementing dense layer logic in OpenCL, the library delegates to the existing, well-tested ANN library. This avoids code duplication and ensures consistency.
Data transfer overhead: After the CNN layers complete (including flatten), the flattened activation vector is copied from GPU memory to CPU memory for ANN processing. During training, the ANN gradient is copied back to GPU memory for CNN backpropagation. These transfers are small (the flattened vector is typically a few hundred to a few thousand elements) compared to the computation savings from running convolutions on GPU.

Worker Model

The GPU implementation uses a coordinator–worker architecture:

  • CoreGPU<T> is the coordinator. It creates workers, distributes training samples across GPUs, and merges gradients after each epoch. It does not own any GPU device context.
  • CoreGPUWorker<T> is the per-device worker. Each worker owns:
    • An OpenCLWrapper::Core instance bound to one GPU device
    • A GPUBufferManager that allocates and manages all OpenCL buffers
    • A GPUKernelBuilder that generates and registers OpenCL kernels
    • An ANN::Core<T> instance for dense layer computation on CPU

Each worker can independently process a subset of training samples. During prediction, samples are distributed evenly across workers for maximum throughput.

Kernel Source Files

OpenCL kernel source code is organized into separate files by function. These files are loaded and compiled at runtime by CoreGPUWorker:

FilePurposeKey Kernels
opencl/CNN_Defines.hpp.cl Type definitions — sets #define TYPE float (or double/int) so all kernels use the correct numeric precision TYPE macro
opencl/CNN_Propagate.cpp.cl Forward propagation kernels for all CNN layer types calculate_conv2d, calculate_relu, calculate_maxpool, calculate_avgpool, calculate_globalavgpool, calculate_globaldualpool
opencl/CNN_Backpropagate.cpp.cl Gradient computation kernels for all CNN layer types calculate_dCost_dFilters, calculate_dCost_dBiases, calculate_dCost_dInput, calculate_dCost_dRelu, calculate_dCost_dMaxpool
opencl/CNN_Update.cpp.cl Parameter update and utility kernels accumulate_gradients, update_parameters, update_parameters_adam, zero_buffer
opencl/CNN_InstanceNorm.cpp.cl Normalization kernels (InstanceNorm and BatchNorm) calculate_norm_mean, calculate_norm_var, calculate_norm_propagate, calculate_norm_dGamma, calculate_norm_dBeta, calculate_norm_dInput
opencl/CNN_Bridge.cpp.cl Data transfer between CNN GPU buffers and ANN CPU buffers bridge_cnn_to_ann, bridge_ann_to_cnn
opencl/CNN_Residual.cpp.cl Residual (skip connection) kernels — identity addition and 1×1 projection residual_add, residual_add_proj, residual_bwd, residual_bwd_proj_dskip, residual_bwd_proj_dw

Type System & Precision

All OpenCL kernels use the TYPE macro defined in CNN_Defines.hpp.cl. This allows the same kernel source to work with different numeric types:

// CNN_Defines.hpp.cl — generated at runtime based on template parameter T
#define TYPE float    // For Core<float>
#define TYPE double   // For Core<double>

The CoreGPUWorker constructor writes this file dynamically based on the template parameter T before compiling kernels. This ensures type safety — if you instantiate Core<double>, all GPU computations use double precision automatically.

Double precision on GPU: Not all GPUs support cl_khr_fp64 (double precision). The library checks for this extension at runtime. If unavailable, GPU mode falls back to an error. For maximum compatibility, use float.

Kernel Builder Pattern

Rather than hardcoding kernel launches, the library uses a GPUKernelBuilder that walks the layer configuration and dynamically generates the correct sequence of kernel calls:

  1. Walk the layer list — iterate over layersConfig.cnnLayers in order
  2. Track shapes — compute output shapes layer-by-layer, maintaining activation buffer offsets
  3. Register kernels — for each layer, register the appropriate kernel with its arguments (buffer references, offsets, dimensions)
  4. Execute in sequence — at runtime, kernels execute in the order they were registered

This pattern means adding a new layer type requires only:

  • Writing the OpenCL kernel (in the appropriate .cl file)
  • Adding a case in GPUKernelBuilder to register it
  • Adding buffer allocation in GPUBufferManager if the layer needs new buffers
Why not a single monolithic kernel? Separate kernels per layer allow the OpenCL runtime to optimize each kernel independently, enable better GPU occupancy (each kernel has different work-item counts), and make debugging easier (you can test each kernel in isolation).
