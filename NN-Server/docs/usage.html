1. Configuration File

NN-Server is configured through a JSON file. Pass the path as the first argument, or it defaults to config.json in the current directory.

./NN-Server config.json      # explicit path
./NN-Server                   # uses ./config.json
FieldTypeRequiredDefaultDescription
"model"stringYes—Path to the trained model JSON file
"port"integerNo8080TCP port to listen on (1–65535)
"poolSize"integerNoCPU coresNumber of Core instances in the pool
"maxBodySize"integerNo10Maximum request body size in megabytes. Requests exceeding this limit are rejected with HTTP 413. Set to 0 for unlimited
"maxQueueSize"integerNo0 (unlimited)Maximum number of concurrent requests. When exceeded, new requests are rejected with HTTP 503
"logFile"stringNo—Path to the log file. When set, each request is logged with date, time, client IP, method, path, status code, and duration
"maxLogSize"integerNo1Maximum log file size in gigabytes. When reached, the log wraps around and overwrites from the beginning. Set to 0 for unlimited
Input/output formats are auto-detected from the model file. The server reads "inputType", "inputShape", "outputType", and "outputShape" directly from the trained model JSON — matching the same fields used by NN-CLI. When "inputType" is "image", the "inputShape" is required. When "outputType" is "image", the "outputShape" is required.

Minimal config.json

{
  "model": "trained_model.json"
}

Full config.json

{
  "model": "/models/isic_checkpoint.json",
  "port": 9090,
  "poolSize": 4,
  "maxBodySize": 20,
  "logFile": "/var/log/nn-server.log",
  "maxLogSize": 2
}

Log format

Each request is logged as a single line:

[2026-03-23 14:30:05.123] 192.168.1.10 POST /predict 200 45.2ms
[2026-03-23 14:30:06.456] 192.168.1.10 GET /health 200 0.3ms
[2026-03-23 14:30:07.789] 10.0.0.5 POST /predict 413 0.1ms

2. HTTP Endpoints

MethodPathDescription
GET/healthHealth check — returns {"status":"ok"}
POST/predictRun prediction on input data
anyotherReturns 404 {"error":"Not found. Use POST /predict"}

GET /health

Returns a simple health check. Use this to verify the server is running and the model is loaded.

curl http://localhost:8080/health

Response: 200 OK

{ "status": "ok" }

POST /predict

Run inference on the loaded model. The input format is determined by the Content-Type header:

  • application/json (default) — JSON body with an "input" array
  • image/* (e.g. image/jpeg, image/png) — raw image bytes in the body

3. Predict with JSON Input

Send a flat array of floats matching the model's input size.

Request

POST /predict HTTP/1.1
Content-Type: application/json

{
  "input": [0.1, 0.5, 0.9, ...]
}
Input size: For ANN models, the array length must match the first layer's numNeurons. For CNN models, it must equal C × H × W from the model's inputShape.

Response — 200 OK

{
  "output": [0.02, 0.85, 0.01, 0.03, 0.01, 0.01, 0.02, 0.01, 0.03, 0.01, 0.00]
}

4. Predict with Image Input

Send raw image bytes (JPEG, PNG, BMP, etc.) directly. The server decodes the image, resizes it to the model's inputShape, normalises pixel values to [0, 1], and converts to NCHW layout.

Request

POST /predict HTTP/1.1
Content-Type: image/jpeg
Content-Length: 32451

<raw JPEG bytes>

With curl

curl -X POST http://localhost:8080/predict \
     -H "Content-Type: image/jpeg" \
     --data-binary @photo.jpg
CNN only: Image input requires a CNN model with a defined inputShape. ANN models do not support image input — use JSON instead.

5. Output Formats

JSON Output (default)

When the model's "outputType" is "vector" (or absent), predictions are returned as a JSON array:

Content-Type: application/json

{
  "output": [0.02, 0.85, 0.01, ...]
}

For classification models with softmax output, the values sum to approximately 1.0 and represent class probabilities.

Image Output

When the model's "outputType" is "image", the "outputShape" field is required. The output vector is reconstructed into a PNG image using that shape:

Content-Type: image/png

<raw PNG bytes>

This is useful for image-to-image models (segmentation, super-resolution, style transfer, etc.).

6. Concurrency Model

NN-Server uses a thread-safe pool of pre-loaded Core objects for concurrent inference:

  1. On startup, NN_SERVER_POOL_SIZE independent copies of the model are loaded into memory
  2. Each incoming HTTP connection is dispatched to QThreadPool
  3. The request handler acquires a Core from the pool (blocks if all are busy)
  4. Prediction runs on the acquired Core
  5. The Core is released back to the pool

This ensures that multiple requests can run truly in parallel (up to NN_SERVER_POOL_SIZE concurrent predictions), with each Core maintaining fully isolated weights and buffers.

Pool sizing: Set NN_SERVER_POOL_SIZE based on available memory and expected concurrency. Each Core holds a full copy of the model weights. For large CNN models, 2–4 is often sufficient.

7. Error Handling

All errors are returned as JSON with an "error" field:

StatusConditionResponse Body
400Malformed HTTP request{"error":"Malformed HTTP request"}
400Invalid JSON body{"error":"Invalid JSON: ..."}
400Missing "input" array{"error":"Request must contain an 'input' array"}
400Missing request body{"error":"Missing request body"}
404Unknown endpoint{"error":"Not found. Use POST /predict"}
413Request body exceeds maxBodySize{"error":"Request body too large. Maximum allowed: N bytes"}
503Queue full (maxQueueSize exceeded){"error":"Server busy. Too many queued requests"}
500Input size mismatch{"error":"Prediction failed: Input size (N) does not match expected shape size (M)"}
500Image decode failure{"error":"Prediction failed: ..."}

Startup errors (missing NN_MODEL_CONFIG, invalid model file, etc.) are printed to stderr and the process exits with code 1.
